# تغييرات الباك‌اند لسير عمل تسليم المشتاح — دليل فريق التطبيق

## 1. السلوك القديم

- المناوبة المنتهية تنشئ تسليم مشتاح بحالة `PENDING`.
- المناوبة الداخلة يمكنها استعراض التسليم المعلّق، لكن لا يوجد إشارة واضحة من الباك‌اند بأن هذا التسليم **يحجب** سير العمل.
- لا يوجد قيد في قاعدة البيانات يمنع وجود أكثر من تسليم معلّق لنفس خط الإنتاج.
- لا يوجد حماية من تنفيذ عمليتي تأكيد/رفض متزامنتين على نفس التسليم.

## 2. المشكلة في السلوك القديم

- يمكن أن تتراكم تسليمات معلّقة بدون حل.
- التطبيق ليس لديه طريقة واضحة لمعرفة أن المستخدم **يجب** أن يتعامل مع التسليم المعلّق قبل أي عمل آخر.
- ظروف التنافس (race conditions) يمكن أن تسمح بإنشاء تسليمين معلّقين لنفس الخط أو تأكيد/رفض نفس التسليم مرتين.

## 3. القاعدة التجارية الجديدة

> المناوبة الداخلة لا يمكنها متابعة العمل الطبيعي على خط إنتاج معين إذا كان هناك تسليم مشتاح معلّق من المناوبة السابقة. يجب حل التسليم المعلّق أولاً.

- كل تسليم معلّق هو **حالة حظر مؤقتة** بين المناوبة المنتهية والمناوبة الداخلة.
- يجب أن ينتقل التسليم خارج حالة `PENDING` فوراً بعد تعامل المناوبة الداخلة معه.

## 4. نطاق التفرّد

**لكل خط إنتاج** — يُسمح بتسليم معلّق واحد فقط لكل `production_line_id` في أي وقت.

هذا يعني:
- خط إنتاج 1 يمكن أن يكون لديه تسليم معلّق.
- خط إنتاج 2 يمكن أن يكون لديه تسليم معلّق آخر في نفس الوقت.
- لكن لا يمكن أن يكون هناك تسليمان معلّقان لنفس الخط.

تم فرض هذا القيد في:
- مستوى الخدمة (service-level check with pessimistic locking)
- مستوى قاعدة البيانات (unique constraint on generated column)

## 5. ما تغيّر في الباك‌اند

### قاعدة البيانات
- Migration جديد (`V16`): عمود مُولَّد `pending_lock` مع قيد فريد يمنع وجود أكثر من تسليم `PENDING` لنفس خط الإنتاج.

### المستودع (Repository)
- طريقتان جديدتان مع قفل تشاؤمي (`SELECT ... FOR UPDATE`):
  - `findByIdForUpdate` — يُستخدم عند التأكيد/الرفض/حل النزاع
  - `findByProductionLineIdAndStatusForUpdate` — يُستخدم عند إنشاء تسليم جديد

### الخدمة (Service)
- `createPendingHandover`: يستخدم القفل التشاؤمي + التقاط `DataIntegrityViolationException` للحماية من التنافس.
- `confirmHandover`: يستخدم `findByIdForUpdate` لمنع التأكيد المتزامن.
- `rejectHandover`: يستخدم `findByIdForUpdate` لمنع الرفض المتزامن.
- `resolveDispute`: يستخدم `findByIdForUpdate` لمنع الحل المتزامن.
- `toPendingHandoverResponse`: يتضمن الآن بيانات الحظر (blocking metadata).

## 6. تغييرات نقاط النهاية (Endpoints)

لا توجد نقاط نهاية جديدة. نفس نقاط النهاية الموجودة:

| Endpoint | Method | الوصف |
|---|---|---|
| `/api/v1/shift-handover` | `POST` | إنشاء تسليم معلّق |
| `/api/v1/shift-handover/pending?lineId={id}` | `GET` | استعلام عن تسليم معلّق لخط إنتاج معين |
| `/api/v1/shift-handover/pending-list` | `GET` | كل التسليمات المعلّقة عبر جميع الخطوط |
| `/api/v1/shift-handover/{id}/confirm` | `POST` | تأكيد التسليم |
| `/api/v1/shift-handover/{id}/reject` | `POST` | رفض التسليم (إنشاء نزاع) |
| `/api/v1/shift-handover/{id}` | `GET` | تفاصيل تسليم محدد |

## 7. تغييرات DTOs / حقول الاستجابة

### `PendingHandoverResponse` — حقول جديدة

```json
{
  "id": 1,
  "productionLineId": 1,
  "productionLineName": "خط الإنتاج 1",
  "outgoingOperatorId": 1,
  "outgoingOperatorName": "أحمد محمد",
  "outgoingShiftType": "MORNING",
  "outgoingShiftDisplayNameAr": "صباحي",
  "status": "PENDING",
  "items": [...],
  "itemCount": 2,
  "totalQuantity": 75,
  "createdAt": "2026-03-31T08:00:00Z",
  "createdAtDisplay": "2026-03-31، 11:00 صباحًا",
  "blocking": true,
  "availableActions": ["CONFIRM", "DISPUTE"],
  "message": "يوجد تسليم مشتاح معلق من المناوبة السابقة. يجب تأكيد أو رفض التسليم قبل المتابعة."
}
```

الحقول الجديدة:

| الحقل | النوع | الوصف |
|---|---|---|
| `blocking` | `boolean` | دائماً `true` للتسليمات المعلّقة. يعني أن سير العمل محظور. |
| `availableActions` | `List<String>` | الإجراءات المتاحة: `["CONFIRM", "DISPUTE"]` |
| `message` | `String` | رسالة عربية لعرضها في حوار الحظر |

### `HandoverResponse` — بدون تغيير

لا توجد تغييرات في `HandoverResponse`. يُرجع كما كان بعد التأكيد أو الرفض.

## 8. أي نقاط نهاية يجب أن يستدعيها التطبيق

### عند دخول خط إنتاج
```
GET /api/v1/shift-handover/pending?lineId={productionLineId}
```

- إذا كانت الاستجابة `data` هي `null` → لا يوجد تسليم معلّق، يمكن المتابعة.
- إذا كانت الاستجابة `data` ليست `null` و `blocking == true` → **يجب حظر** سير العمل وعرض حوار التسليم.

### بديل: عند تسجيل الدخول
```
GET /api/v1/shift-handover/pending-list
```

يُرجع قائمة بكل التسليمات المعلّقة عبر جميع الخطوط. مفيد إذا أراد التطبيق عرض نظرة عامة بعد تسجيل الدخول.

## 9. كيف يكتشف التطبيق وجود تسليم معلّق يحظر العمل

1. استدعِ `GET /api/v1/shift-handover/pending?lineId={id}`.
2. تحقق من `data`:
   - `null` → لا يوجد حظر، تابع العمل الطبيعي.
   - غير `null` → تحقق من `data.blocking`:
     - `true` → **يجب** عرض حوار حظر وإجبار المستخدم على اختيار إجراء قبل المتابعة.

## 10. كيف يمنع التطبيق المستخدم من المتابعة

عند اكتشاف `blocking == true`:
1. اعرض حوار حظر (modal / شاشة كاملة) **لا يمكن إغلاقه**.
2. اعرض بيانات التسليم: اسم المشغّل المنتهي، المناوبة، عدد القطع، تفاصيل المنتجات.
3. اعرض زرين فقط:
   - **تأكيد** (CONFIRM): المناوبة الداخلة توافق على صحة البيانات.
   - **رفض** (DISPUTE): المناوبة الداخلة ترفض البيانات وتنشئ نزاعاً للإدارة.
4. لا تسمح بأي عمل آخر (إنشاء بالتات، طباعة، إلخ) حتى يتم حل التسليم.
5. استخدم حقل `message` كنص رئيسي في الحوار.
6. استخدم `availableActions` لتحديد الأزرار المعروضة.

## 11. طلبات التأكيد والرفض

### التأكيد
```http
POST /api/v1/shift-handover/{handoverId}/confirm
Content-Type: application/json

{
  "incomingOperatorId": 2
}
```

### الرفض (إنشاء نزاع)
```http
POST /api/v1/shift-handover/{handoverId}/reject
Content-Type: application/json

{
  "incomingOperatorId": 2
}
```

`incomingOperatorId` هو معرّف المشغّل الحالي (المناوبة الداخلة) الذي يتعامل مع التسليم.

## 12. ماذا يُرجع الباك‌اند بعد كل إجراء

### بعد التأكيد
```json
{
  "success": true,
  "data": {
    "id": 1,
    "status": "CONFIRMED",
    "incomingOperatorId": 2,
    "incomingOperatorName": "محمد علي",
    "incomingShiftType": "EVENING",
    "confirmedAt": "2026-03-31T14:05:00Z",
    ...
  }
}
```

### بعد الرفض
```json
{
  "success": true,
  "data": {
    "id": 1,
    "status": "DISPUTED",
    "incomingOperatorId": 2,
    "incomingOperatorName": "محمد علي",
    "incomingShiftType": "EVENING",
    "disputedAt": "2026-03-31T14:05:00Z",
    ...
  }
}
```

في كلتا الحالتين:
- الحالة لم تعد `PENDING`.
- التطبيق يمكنه إغلاق حوار الحظر والسماح بالعمل الطبيعي.

### في حالة خطأ (التسليم تم التعامل معه مسبقاً)
```json
{
  "success": false,
  "error": {
    "code": "HANDOVER_ALREADY_RESOLVED",
    "message": "Handover has already been confirmed"
  }
}
```

HTTP Status: `409 Conflict`

## 13. تغييرات واجهة المستخدم / سير العمل المطلوبة في التطبيق

1. **عند اختيار خط إنتاج**: استدعِ `/pending?lineId=X` قبل السماح بالعمل.
2. **إذا وُجد تسليم معلّق**: اعرض حوار حظر إلزامي.
3. **حوار الحظر**:
   - عنوان: حقل `message` من الاستجابة
   - تفاصيل: اسم المشغّل، المناوبة، عدد القطع، تفاصيل المنتجات
   - أزرار: `availableActions` → زر "تأكيد" وزر "رفض"
4. **بعد التأكيد أو الرفض**: أغلق الحوار، تابع العمل الطبيعي.
5. **إذا فشل الطلب بـ `409`**: أعد استدعاء `/pending?lineId=X`:
   - إذا كانت النتيجة `null` → تم حل التسليم من مكان آخر، تابع العمل.
   - إذا كانت النتيجة غير `null` → أعد عرض الحوار (حالة نادرة).

## 14. حالات الحافة التي يجب التعامل معها

| الحالة | كيفية التعامل |
|---|---|
| استدعاء `/pending?lineId=X` يُرجع `null` | لا يوجد حظر، تابع |
| مستخدمان يحاولان تأكيد نفس التسليم | الثاني يحصل على `409 HANDOVER_ALREADY_RESOLVED` |
| مستخدم يحاول تأكيد تسليم تم رفضه | يحصل على `409 HANDOVER_ALREADY_RESOLVED` |
| إنشاء تسليم جديد بينما يوجد معلّق | يحصل على `409 PENDING_HANDOVER_EXISTS` |
| التطبيق offline ثم يعود | أعد استدعاء `/pending?lineId=X` عند العودة |
| التسليم حُلّ بينما الحوار مفتوح | عند إرسال التأكيد/الرفض، إذا جاء `409`، أعد الاستعلام |

## 15. ملاحظات الترحيل / التوافقية

- **لا توجد نقاط نهاية جديدة** — نفس المسارات والطرق.
- **حقول جديدة في `PendingHandoverResponse`**: `blocking`, `availableActions`, `message`. هذه حقول إضافية ولن تكسر عملاء JSON الحاليين (backward compatible).
- **Migration V16**: يعمل تلقائياً عند بدء التشغيل. لا يحتاج تدخل يدوي.
- **لا يوجد تغيير في المصادقة** — نفس JWT مع دور `PALLETIZER`.

## 16. سيناريوهات اختبار التطبيق المُوصى بها

1. **الحالة العادية بدون تسليم معلّق**: اختر خط إنتاج → لا يظهر حوار → يمكن العمل مباشرة.
2. **وجود تسليم معلّق**: اختر خط إنتاج فيه تسليم معلّق → يظهر حوار حظر → لا يمكن إغلاقه.
3. **التأكيد**: في حوار الحظر، اضغط تأكيد → التسليم ينتقل إلى `CONFIRMED` → الحوار يُغلق → يمكن العمل.
4. **الرفض**: في حوار الحظر، اضغط رفض → التسليم ينتقل إلى `DISPUTED` → الحوار يُغلق → يمكن العمل.
5. **تعامل مزدوج**: حاول تأكيد تسليم تم تأكيده → تأكد من معالجة خطأ `409` بشكل صحيح.
6. **خطوط مختلفة**: تأكد أن تسليم معلّق على خط 1 لا يمنع العمل على خط 2.
7. **حقول الحظر**: تحقق من وجود `blocking=true` و `availableActions` و `message` في استجابة `/pending`.
8. **انقطاع الشبكة**: عند فقدان الاتصال وعودته، يُعاد الاستعلام عن التسليم المعلّق.
9. **إنشاء تسليم مكرر**: حاول إنشاء تسليم جديد على خط فيه تسليم معلّق → تأكد من خطأ `409`.
